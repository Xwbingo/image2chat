# 图片生成来源、引用区遮挡与 CORS 选择设计

## 背景

当前应用存在三个相互独立但范围较小的使用问题：

1. 图片消息没有记录生成时使用的中转站，切换、改名或删除中转站后无法可靠判断图片来源。
2. 引用图片会增加固定发送区的高度，而聊天列表使用固定底部留白，导致会话末尾的图片可能被发送区遮挡。
3. 密钥管理中的 CORS 配置是自由输入框，不便于选择常用配置，也不清楚空值代表直接连接。

本次改动保持现有图片引用流程、最多三张引用图片、引用顺序和逐次文件选择行为，不新增历史图片选择器，也不重构生成接口。

## 目标

- 在生成中、成功和失败的图片消息上显示请求发起时的中转站名称。
- 确保固定发送区高度变化后，最后一条聊天记录仍可滚动到发送区上方。
- 在不干扰用户阅读历史消息的前提下，让已处于底部附近的聊天视图继续贴底。
- 将每个中转站的 CORS 配置改为“直接连接”、“/api/cors”和“自定义”三种选择。
- 保持现有数据库和 API 客户端对 `corsProxy` 最终字符串的处理方式。

## 非目标

- 不为旧图片推测中转站。
- 不显示或保存 API Key、CORS 地址等敏感或无关的请求来源信息。
- 不新增图片历史选择器。
- 不改变图片引用数量、排序语义或上传方式。
- 不新增 Playwright、端到端测试或浏览器自动化测试。
- 不启动开发服务器进行验证。

## 方案选择

采用针对性修复：

- 在图片消息中保存中转站名称快照，而不是渲染时关联可变的中转站配置。
- 动态测量固定底部区域，而不是继续增加固定底部留白。
- 使用受控下拉和条件自定义输入框，而不是可编辑组合框或继续使用单一文本框。

该方案能保证历史来源准确、彻底解决可变高度遮挡，并且不扩大现有数据层和请求层的职责。

## 设计

### 1. 图片生成来源

`Message` 增加可选字段 `providerName?: string`，用于保存中转站名称快照。字段保持可选，以兼容 IndexedDB 中的旧消息，不需要数据迁移。

`useGenerate` 在解析出当前会话使用的 `ProviderPreset` 后，创建助手占位消息时立即写入 `provider.name`。该消息之后只更新生成状态、图片、耗时和错误信息，因此生成中、成功和失败状态始终使用同一个请求时快照。

`MessageBubble` 在图片结果消息的元信息区域显示中转站名称。显示规则如下：

- 生成中：显示名称。
- 生成成功：与尺寸、大小和耗时等元信息一起显示名称。
- 生成失败：显示名称，并保留现有错误与诊断入口。
- 旧消息没有快照：完全省略该字段，不显示“未知”，也不使用当前会话中转站推测。

只保存显示名称，不在消息中复制 Base URL、API Key、CORS 配置或完整中转站对象。中转站之后被改名或删除，不影响历史图片显示。

### 2. 引用图片后的底部布局

`ChatView` 继续负责聊天滚动容器和固定底部区域。固定区域包含 `StatusBar` 与 `Composer`，其实际高度由 `ResizeObserver` 监听。

测量值用于设置聊天列表的动态底部留白，替代当前固定的 `9rem` 假设。正常情况下，底部留白等于固定区域实际高度加 `1rem` 消息间距；固定区域的测量值已经包含安全区内边距，不再重复叠加安全区高度。发送区因为引用图片预览、文本框增长、状态栏或响应式布局而改变高度时，留白同步更新，确保最后一条消息能够完整滚动到发送区上方。

高度变化前记录用户是否位于底部附近。高度更新后：

- 用户原本位于底部附近：将视图保持在底部，使最新图片和“引用”按钮可见。
- 用户正在阅读较早消息：不主动滚动，避免打断阅读；仅更新可滚动范围。

“底部附近”定义为更新前 `scrollHeight - scrollTop - clientHeight <= 80px`，避免要求滚动位置完全相等，并兼容子像素、移动端视口和图片异步布局。

组件卸载时断开 observer。测试环境或浏览器不支持 `ResizeObserver` 时，继续使用当前 `calc(9rem + env(safe-area-inset-bottom))` 基础留白，页面仍可使用，但不提供动态高度增强。

本次不改变以下行为：

- 用户仍从图片消息上的“引用”按钮添加引用。
- Composer 仍显示引用图片预览与顺序。
- 拖动排序仍直接改变请求中的图片顺序。
- 最多引用三张图片。

### 3. CORS 下拉配置

每个中转站的 CORS 控件改为受控选择，包含：

1. `直接连接`：默认选项，最终保存 `undefined`。
2. `/api/cors`：最终保存字符串 `/api/cors`。
3. `自定义`：显示额外文本输入框，保存用户输入的地址或相对路径。

现有值初始化时按以下规则映射：

- 空值或纯空白：直接连接。
- 精确值 `/api/cors`：`/api/cors` 选项。
- 其他非空值：自定义，并将原值填入输入框。

新建和未配置的中转站默认选择直接连接。`/api/cors` 是预置选项，不是默认启用值。

保存时对自定义值去除首尾空格。自定义输入为空时按直接连接保存，避免持久化无意义空字符串。切换选项不立即写入 IndexedDB，仍由现有“保存”按钮统一提交。

密钥测试继续使用当前尚未保存的 API Key 与 CORS 草稿：

- 直接连接传入 `undefined`。
- `/api/cors` 传入该固定字符串。
- 自定义传入裁剪后的文本。

底层 `applyCorsProxy`、生成请求和远程图片下载逻辑不需要修改接口。

## 组件和数据流

### 生成流程

1. `useGenerate` 读取会话与中转站。
2. 创建用户消息。
3. 创建带中转站名称快照的助手图片消息。
4. 使用同一中转站执行请求。
5. 更新同一助手消息为成功或失败状态。
6. `MessageBubble` 从消息快照渲染来源。

### 布局流程

1. `ChatView` 挂载固定底部区域引用。
2. `ResizeObserver` 报告区域高度。
3. `ChatView` 更新消息列表底部留白。
4. 如果更新前视图临近底部，则在布局更新后保持贴底。
5. 如果不临近底部，仅扩大可滚动范围。

### CORS 配置流程

1. `SettingsSheet` 从 `ProviderPreset.corsProxy` 推导选择模式和自定义草稿。
2. 用户选择模式，必要时填写自定义值。
3. “测试”将当前草稿转换为最终 CORS 值后调用现有验证函数。
4. “保存”执行相同转换并更新中转站记录。
5. 后续生成请求继续读取持久化的 `corsProxy`。

## 错误与兼容处理

- 旧消息没有中转站字段时正常渲染，不迁移、不猜测。
- 中转站名称为空时不渲染空标签。
- 不支持 `ResizeObserver` 时使用基础留白降级。
- 自定义 CORS 空值按直接连接处理。
- 现有非 `/api/cors` 配置无损映射到自定义模式。
- 本次不新增 CORS URL 格式限制，继续接受现有支持的相对路径和 HTTP(S) 地址，避免破坏已有配置。

## 测试策略

测试以现有 Vitest 单元测试和 React 组件测试为主，不安装或运行 Playwright，也不启动项目。

### 图片来源

- `useGenerate` 创建助手消息时保存请求时的中转站名称。
- 成功、失败更新不覆盖名称快照。
- `MessageBubble` 在生成中、成功和失败状态显示名称。
- 没有快照的旧消息不显示来源。
- 切换或修改中转站后，已保存消息仍显示原名称。

### 底部布局

- `ChatView` 使用测得高度更新列表底部留白。
- 引用预览导致底部区域增高时同步更新留白。
- 原本临近底部时，高度变化后保持贴底。
- 不在底部附近时，不强制滚动。
- observer 在卸载时清理。
- `ResizeObserver` 不可用时使用基础留白。

### CORS 配置

- 空值初始化为直接连接。
- `/api/cors` 初始化为固定选项。
- 其他已有值初始化为自定义并保留文本。
- 三种模式分别转换为 `undefined`、`/api/cors` 和裁剪后的自定义值。
- 自定义空值按直接连接保存。
- 密钥测试使用当前未保存的模式和值。
- 保存后持久化正确值。

## 验收标准

- 新生成图片从生成中到成功或失败均显示生成时的中转站名称。
- 中转站改名、切换或删除不会改变已有图片显示的来源。
- 旧图片不显示错误或推测的来源。
- 添加、删除或排序引用图片后，聊天末尾能够完整滚动到发送区上方。
- 临近底部的用户继续看到最新内容，阅读旧消息的用户不被强制拉回底部。
- CORS 控件提供直接连接、`/api/cors` 和自定义三个选项，默认直接连接。
- 已有自定义 CORS 配置能够正确回显、测试和保存。
- 所有相关单元测试、完整单元测试、TypeScript 检查和生产构建通过。
